# How's the Texture Atlas work going?

I've definitely been making some steady progress on many tools at Dot-Stuff that will be very helpful with a lot of things Adobe Flash related. Mostly for videogame asset development and in general just nice ways to interact with animation data from Flash.

One of these projects projects is a full FLA file decoder and encoder I've been working on in Haxe! Initially it was going to be a normal XFL decoder (which there are many of already), but with time I started digging around some documentation and decided to make it a full FLA decoder that includes reading and writing FLAs generated in Flash CS4, prior to the XFL switch.

Now, onto the big boys, flixel-animate and BetterTextureAtlas. My work for them as of recently has been mostly bugfix/util features related, not many major changes. Part of the reason for this is because, for flixel-animate, many of the more "breaking" changes have been developed on a seperate experimental branch which will be merged eventually once I decide the library is ready for it's big update. Which will include many new features and an overhauled "draw inheritance" system which will give the user much more freedom on how symbols get rendered on screen.

On the other side, BetterTextureAtlas dev has been a lot more quiet since many of its planned overhauls are still private and with a lot of room for change. The reason for this is because the rewrite will be done in Haxe, which also requires some custom tools I've been working on that help develop plugins for Flash using Haxe. While that rewrite keeps cooking in the oven, as I mentioned publicly before, BTA will continue to recieve bugfixes and some minor features where possible.

For both of these projects I want to be very receptive for issues people might run into so PLEASE, if you use our tools, dont be scared to drop a DM or create a Github issue for us to resolve. I want to make these tools the best they can possibly be for everyone who decides to take the time to try them! :)
